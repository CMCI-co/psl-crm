Wiring Training Modules to Supabase

Everything you click in the prototype today runs on in-browser mock data. This guide is the checklist to make it real — a database, login, file storage, and the permission rules that keep authoring locked to Campus Directors and National Staff. No code changes are needed to read it; do the steps in order and you'll have a working backend.

Who this is for. You (or whoever sets up the account). You'll spend most of the time pasting SQL into Supabase's editor and clicking buttons — not programming. Budget about an hour for a first pass.

1

Create the Supabase project

  1. Go to supabase.com, sign in, and click New project.
  2. Name it psl-crm, pick a region close to your members (e.g. East US), and set a strong database password — save it in your password manager.
  3. When it finishes provisioning, open Project Settings → API and copy two values: the Project URL and the anon public key. These are what the app uses to talk to Supabase. (The anon key is safe to ship in the frontend — the security comes from the rules in Step 4.)
2

Create the database tables

Open SQL Editor → New query, paste the block below, and click Run. It creates the whole schema: people and their roles, modules, the slides inside them, the test questions, and a record of every attempt.

-- 1. People. Mirrors Supabase Auth, adds a role per person.
create type user_role as enum ('national_staff', 'campus_director', 'member', 'candidate', 'applicant', 'alumni');

create table profiles (
  id          uuid primary key references auth.users(id) on delete cascade,
  full_name   text,
  role        user_role not null default 'member',
  campus      text,
  created_at  timestamptz default now()
);

-- 2. Modules. The top-level training unit.
create table modules (
  id           uuid primary key default gen_random_uuid(),
  title        text not null,
  collection   text,
  audience     text[] default '{}',   -- e.g. {Candidates,Members}
  pass_score   int  default 75,
  status       text default 'draft',   -- draft | published
  created_by   uuid references profiles(id),
  created_at   timestamptz default now(),
  updated_at   timestamptz default now()
);

-- 3. Slides. One row per slide; `content` holds the type-specific data.
create table slides (
  id         uuid primary key default gen_random_uuid(),
  module_id  uuid references modules(id) on delete cascade,
  position   int  not null,
  type       text not null,        -- text | image | video | game
  content    jsonb not null default '{}'  -- title, body, pairs, cards, etc.
);

-- 4. Test questions for a module's final test.
create table questions (
  id         uuid primary key default gen_random_uuid(),
  module_id  uuid references modules(id) on delete cascade,
  position   int,
  prompt     text not null,
  options    text[] not null,
  correct    int not null     -- index of the right option
);

-- 5. Attempts. Every time a learner takes the test.
create table attempts (
  id          uuid primary key default gen_random_uuid(),
  module_id   uuid references modules(id) on delete cascade,
  user_id     uuid references profiles(id) on delete cascade,
  score       int  not null,
  passed      boolean not null,
  taken_at    timestamptz default now()
);

Why content jsonb? Each game type stores different fields — a match game has pairs, flashcards have cards, the sorting game has buckets and items. One flexible content column holds all of them, which matches exactly how the builder already shapes each slide. No schema change when you add a new game type later.

3

Turn on login & assign roles

Open Authentication → Providers. Enable Email (magic-link is the simplest — members get a sign-in link, no passwords). If your chapters use Google Workspace, enable Google too.

A person's role is what unlocks editing. When someone signs up, create their profile row and set the role. Paste this once so new sign-ups automatically get a profile:

-- Auto-create a profile row whenever someone signs up.
create function handle_new_user() returns trigger as $$
begin
  insert into profiles (id, full_name)
  values (new.id, new.raw_user_meta_data->>'full_name');
  return new;
end; $$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function handle_new_user();

To make someone a leader, run one line (or build a tiny admin screen later):

update profiles set role = 'campus_director'
where id = '<their-user-id>';
4

Lock down who can edit (Row Level Security)

This is the most important step — it's what enforces the exact rule from the prototype: anyone can read published modules, but only Campus Directors and National Staff can create or change them. These rules run on Supabase's servers, so they can't be bypassed from the browser.

-- Helper: is the current user a module editor?
create function is_editor() returns boolean as $$
  select exists (
    select 1 from profiles
    where id = auth.uid()
      and role in ('campus_director', 'national_staff')
  );
$$ language sql security definer;

alter table modules   enable row level security;
alter table slides    enable row level security;
alter table questions enable row level security;
alter table attempts  enable row level security;

-- Everyone signed in can read published modules…
create policy "read published" on modules for select
  using (status = 'published' or is_editor());

-- …but only editors can write.
create policy "editors write" on modules for all
  using (is_editor()) with check (is_editor());

-- Slides & questions follow the same editor rule.
create policy "slides editors" on slides for all
  using (is_editor()) with check (is_editor());
create policy "slides read" on slides for select using (true);
create policy "q editors" on questions for all
  using (is_editor()) with check (is_editor());
create policy "q read" on questions for select using (true);

-- A learner can record & read their own attempts; editors see all.
create policy "own attempts" on attempts for all
  using (user_id = auth.uid() or is_editor())
  with check (user_id = auth.uid());

Don't skip RLS. Until you run these, every table is either fully locked or fully open. With them in place, the Editing as Campus Director badge and the locked "authoring restricted" state in the prototype become real, server-enforced behavior.

5

Set up file storage for video & images

The image and video slides need somewhere to keep the actual files. Open Storage → New bucket and create:

BucketPublic?Holds
module-mediaPublic readSlide images & lesson videos
coversPublic readModule cover images

Mark them public for reading (so learners can watch without a signed URL each time), then add an upload rule so only editors can put files in:

create policy "editors upload media" on storage.objects
  for insert with check (bucket_id = 'module-media' and is_editor());

Large videos. Supabase storage handles files up to 5 GB on paid plans. For long lessons, many chapters host the video on YouTube/Vimeo (unlisted) and store just the link in the slide's content — cheaper and faster. Either works with the current video slide.

6

Connect the prototype to Supabase

This is the one part that touches the app code. Add the Supabase client library and point it at your project:

// add to the page <head>
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>

// initialize once
const db = supabase.createClient(
  'https://YOUR-PROJECT.supabase.co',
  'YOUR-ANON-KEY'
);

Then swap the four mock-data touchpoints for live calls. The shapes already line up with the builder & player, so this is mostly find-and-replace:

In the prototypeReplace with
Library list (MODULES)db.from('modules').select()
Builder autosavedb.from('modules' / 'slides').upsert(...)
Role gate (Tweaks dropdown)the signed-in user's profiles.role
Test submit / passdb.from('attempts').insert(...)

The role gate is already built right. Today the prototype reads a role from the Tweaks panel; in production you read it once from profiles.role after login and pass it into <TrainingBuilder role={…}> and <TrainingLibrary role={…}> exactly as they're wired now. The confetti, the pass/fail test, the slide types — none of that changes.

7

Go-live checklist

  • Seed a real module — paste the History & Heritage content (or rebuild it in the live builder once connected) so there's something to take on day one.
  • Make yourself National Staff, then invite one Campus Director and confirm they can edit but a test "member" account cannot.
  • Take the test as a member end-to-end and confirm an attempts row appears with the right score.
  • Turn on email confirmations in Auth settings before inviting real members.
  • Set a custom domain for auth emails (Auth → Email Templates) so links come from your chapter, not a generic Supabase address.

Questions on any step — especially Step 4 (the permission rules) or Step 6 (wiring the app) — are the ones to bring back to me. I can adjust the prototype's data layer to match whatever you stand up.
