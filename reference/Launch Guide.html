Create the Supabase project

  1. At supabase.com, click New project. Name it psl-crm, choose a US region, set a strong database password (save it).
  2. Open Project Settings → API and copy the Project URL and the anon public key — the app uses these to connect. (The anon key is safe in the frontend; security comes from Step 5.)
  3. Keep the service_role key secret — it bypasses all rules and is only used server-side, in Step 7.

The complete database schema

Open SQL Editor → New query and run each block below in order. They build the whole system. (You can paste them all at once.)

3a · People & roles

create type journey_stage as enum ('applicant','candidate','member','alumni');
create type user_role    as enum ('national_staff','campus_director','leader','member');

-- Every person in the chapter, at any stage. Login is optional:
-- applicants have a profile but no auth account yet (auth_id is null).
create table profiles (
  id          uuid primary key default gen_random_uuid(),
  auth_id     uuid references auth.users(id) on delete set null,
  first_name  text, last_name text,
  stage       journey_stage not null default 'applicant',
  role        user_role,                -- null for non-staff
  email       text, phone text, hometown text,
  school      text, major text, grad_year text,
  cohort      text, campus text, member_id text,
  status      text default 'active',
  created_at  timestamptz default now()
);

3a-ii · Leadership roles (rides the profile through every stage)

A leadership role is its own row tied to the person — never an edit that overwrites the last one. The current flag (really just "no end date yet") drives whether it shows as an active tenure or a permanent historic label, and because it hangs off profiles it keeps showing after the person moves Member → Alum.

create table leadership_roles (
  id          uuid primary key default gen_random_uuid(),
  profile_id  uuid references profiles(id) on delete cascade,
  title       text not null,        -- President, Treasurer, Chaplain, Alumni Board Chair…
  scope       text default 'chapter', -- chapter | alumni
  started_on  date, ended_on date,        -- ended_on null = currently serving
  term_label  text                        -- '2025–26' for display
);
-- "current officers" is just: ended_on is null.
-- "has ever held office" (Member→Alum) is: any row exists for that profile.
create view current_officers as
  select p.*, l.title, l.term_label
  from profiles p join leadership_roles l on l.profile_id = p.id
  where l.ended_on is null;

This is exactly what the prototype now shows. The Profile Card lists current offices (gold) above permanent "Former <title>" labels, and the Directory's Leadership filter works in both the Members and the Alumni views — because the same rows survive the stage change. To end a tenure, you set ended_on; the role instantly becomes a historic label without losing the history.

create table applications (

  id          uuid primary key default gen_random_uuid(),

  interviewer uuid references profiles(id),

  overall     numeric,
  recommend   text,                     -- advance|discuss|hold
  notes       text,
  taken_at    timestamptz default now()
);

create table modules (
  id uuid primary key default gen_random_uuid(),

);

  id uuid primary key default gen_random_uuid(),

create table attempts (
  id uuid primary key default gen_random_uuid(),
  module_id uuid references modules(id) on delete cascade,

create table resources (
  id uuid primary key default gen_random_uuid(),

3d · Relationship tracker & engagement

-- Who owns each alum relationship + the cadence/health read.
create table relationships (
  profile_id uuid primary key references profiles(id) on delete cascade,
  owner_id uuid references profiles(id),
  cadence text, last_contact date, next_due date,
  health text, open_to text[] default '{}'   -- {connect,mentor,give}
);
-- Every logged touch (call/text/email/visit).
create table interactions (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references profiles(id) on delete cascade,
  by_id uuid references profiles(id),
  channel text, note text, areas text[], happened_at date default now()
);
-- Assigned follow-up tasks.
create table followups (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references profiles(id) on delete cascade,
  owner_id uuid references profiles(id),
  title text, due date, channel text, priority text,
  status text default 'todo'
);
-- Life-area health, prayer requests, giving (the engagement record).
create table life_updates (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references profiles(id) on delete cascade,
  area text, status text, tone text, note text,
  by_id uuid references profiles(id), updated_at date default now()
);
create table prayer_requests (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references profiles(id) on delete cascade,
  text text, open boolean default true, created_at date default now()
);
create table giving (
  id uuid primary key default gen_random_uuid(),
  profile_id uuid references profiles(id) on delete cascade,
  amount numeric, frequency text, since text, lifetime numeric
);

That's the whole burrito. Eleven tables cover every screen in the prototype. Notice everything points back to profiles — that's what lets a single person carry their application, interview, training, and engagement history from applicant to alum.

Logins & the role system

Open Authentication → Providers and enable Email (magic-link — members sign in with a link, no passwords). Add Google if your chapters use Workspace.

When a staff/member account signs in, link it to their profile and stamp a role. This trigger does it automatically:

create function link_profile() returns trigger as $$
begin
  -- match by email to an existing profile, else create one
  update profiles set auth_id = new.id
    where email = new.email and auth_id is null;
  if not found then
    insert into profiles (auth_id, email, first_name, stage, role)
    values (new.id, new.email, new.raw_user_meta_data->>'name', 'member', 'member');
  end if;
  return new;
end; $$ language plpgsql security definer;

create trigger on_auth_signup after insert on auth.users
  for each row execute function link_profile();

Promote a leader with one line:

update profiles set role = 'campus_director' where email = 'you@chapter.org';
5

Security rules — who sees & does what

Row Level Security runs on Supabase's servers, so these rules can't be bypassed from a browser. They enforce exactly the permissions in the prototype: leaders manage everything; members see what's meant for them; applicants' data is staff-only.

-- helpers
create function my_role() returns user_role as $$
  select role from profiles where auth_id = auth.uid();
$$ language sql security definer stable;

create function is_staff() returns boolean as $$
  select my_role() in ('campus_director','national_staff','leader');
$$ language sql security definer stable;

-- turn RLS on for everything
alter table profiles, applications, interviews, modules, slides,
  questions, attempts, resources, relationships, interactions,
  followups, life_updates, prayer_requests, giving enable row level security;

-- staff can do anything on the working tables
create policy staff_all on profiles     for all using (is_staff()) with check (is_staff());
create policy staff_all on applications for all using (is_staff()) with check (is_staff());
create policy staff_all on interviews   for all using (is_staff()) with check (is_staff());
-- (repeat staff_all for relationships, interactions, followups,
--  life_updates, prayer_requests, giving)

-- a member can read & edit their OWN profile
create policy own_profile on profiles for all
  using (auth_id = auth.uid()) with check (auth_id = auth.uid());

-- training: everyone reads published modules; staff write
create policy read_pub on modules for select using (status='published' or is_staff());
create policy staff_w  on modules for all using (is_staff()) with check (is_staff());
create policy own_att  on attempts for all
  using (user_id = (select id from profiles where auth_id=auth.uid()) or is_staff())
  with check (true);

Applicant data is staff-only by default. Because applications and interviews only have the staff_all policy, no member can read another person's application or interview scores. That's the behavior you want — and it's enforced by the database, not just hidden in the UI.

Why an Edge Function (not a direct insert)

You could let the public form insert straight into Supabase with the anon key, but a small server endpoint is worth it: it validates the data, blocks spam, sends the confirmation email, and never exposes your tables to the open internet. Supabase Edge Functions are built for exactly this.

The intake function

In the Supabase dashboard, Edge Functions → Deploy a new function named submit-application, with this body. It uses the secret service-role key (server-side only) so it can write even though the public has no table access:

import { createClient } from 'jsr:@supabase/supabase-js@2';

Deno.serve(async (req) => {
  const cors = { 'Access-Control-Allow-Origin': 'https://yourchapter.org',
                'Access-Control-Allow-Headers': 'content-type' };
  if (req.method === 'OPTIONS') return new Response('ok', { headers: cors });

  const form = await req.json();   // the website's form fields
  const db = createClient(
    Deno.env.get('SUPABASE_URL'),
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')   // secret, server-only
  );

  // 1) create the person as an applicant
  const { data: profile } = await db.from('profiles').insert({
    first_name: form.firstName, last_name: form.lastName,
    email: form.email, phone: form.phone, hometown: form.hometown,
    school: form.school, major: form.major, grad_year: form.gradYear,
    stage: 'applicant'
  }).select().single();

  // 2) attach their application
  await db.from('applications').insert({
    profile_id: profile.id, term: form.term, sponsor: form.sponsor,
    gpa: form.gpa, answers: form.answers, references_: form.references
  });

  return new Response(JSON.stringify({ ok: true }),
    { headers: { ...cors, 'content-type': 'application/json' } });
});

Point your website form at it

However your site is built (Squarespace, WordPress, Webflow, custom), the form's submit handler does one fetch to the function URL:

await fetch('https://YOUR-PROJECT.functions.supabase.co/submit-application', {
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify({
    firstName, lastName, email, phone, hometown, school, major,
    gradYear, term, sponsor, gpa,
    answers: [{ q: 'Why join?', a: essay1 }, /* … */],
    references: [{ name: ref1Name, relation: ref1Rel }]
  })
});

The result. The moment someone submits, a profile with Applicant stage and its application exist in the database. Open the Directory → Applicants queue and the new Applicant Card is there. Click it and you're on the exact Application Review screen from the prototype — read their answers, then Start interview to score them. Advancing to Candidate just flips stage; the same record carries forward, no re-typing.

Nice extras once it works. Have the function also email the applicant a confirmation, notify the chapter's staff channel, and create a "Review new applicant" follow-up task. All three are a few more lines in the same function.
